what_is_islo

Never heard of islo.dev? Sixty-second briefing.

Islo is a cloud sandbox platform for AI coding agents from Incredibuild Software Ltd. Instead of letting Claude, Codex, or Cursor agents loose on your laptop — or inside a flaky docker-in-docker container — Islo gives each agent a persistent, isolated “computer”: a real Linux VM where Postgres, Redis, your API, and your dev services run on localhost, 24/7.

The part that makes security teams exhale: every agent sits behind a security gateway. The agent only ever sees placeholder credentials; real secrets are injected at the network egress, with a per-request audit trail. Oversized API responses get trimmed before they hit the agent’s context, and deny policies block risky or exfiltration-shaped requests outright.

12:04:01 ALLOW POST api.stripe.com/v1/charges secret:injected@egress audit:✓ 12:04:07 TRIM GET api.github.com/repos/…/issues response 2.1MB → 64KB audit:✓ 12:04:13 DENY POST paste-bin.example/upload policy:exfiltration audit:✓

Security gateway

Agents never hold real credentials. Secrets swap in at egress, per-request audit, deny policies for banned hosts and exfiltration patterns.

Real persistent environments

Actual Linux VMs — “not docker-in-docker.” Services, disk, and git state stay warm between sessions.

related_tooling

Pairs well with crabbox.sh.

🦀 Warm a box, sync the diff, run the suite.

Crabbox is an open-source (MIT) remote test-execution control plane from the OpenClaw project: lease a throwaway cloud machine, sync your uncommitted working tree, run the command remotely, stream the output back, release. Built for the “too many agents, too many test suites, one very tired Mac” problem.

The connection: islo.dev is one of crabbox’s delegated-run providers. Point crabbox at Islo with an ISLO_API_KEY and it hands the whole sandbox lifecycle to Islo’s API — no SSH, no broker. Your Islo credits (hello, YOSSI150) power the boxes underneath.
